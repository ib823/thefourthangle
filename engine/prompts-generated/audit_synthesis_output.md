Spacing tokens. The CSS uses raw pixel values (16px, 22px, 30px) with no design tokens for spacing. Define: --space-1: 4px, --space-2: 8px, --space-3: 12px, --space-4: 16px, --space-5: 20px, --space-6: 24px, --space-8: 32px, --space-10: 40px, --space-12: 48px, --space-16: 64px. Then replace every raw pixel value in padding, margin, and gap declarations with a token reference. This is the single biggest maintainability gap — every spacing change today requires a find-and-replace guessing game instead of adjusting one variable.
Typography scale tokens. The site has --text-home-title-fluid (clamp-based) for the homepage title, but no systematic type scale tokens for the other 8 size steps. Define: --text-xs: 0.6875rem (11px), --text-sm: 0.75rem (12px), --text-base: 0.875rem (14px), --text-md: 1rem (16px), --text-lg: 1.125rem (18px), --text-xl: 1.5rem (24px), --text-2xl: 2.5rem (40px), --text-3xl: 3.375rem (54px). Each should have a paired line-height token: --leading-xs: 1.5, --leading-base: 1.6, --leading-display: 1.15. Currently, developers must remember that 24px headlines use 34.8px line-height while 54px headlines use 62.1px — this is fragile knowledge that should live in the system, not in people's heads.
Motion tokens. All 7 transition signatures use hardcoded 0.15s ease. Define: --duration-fast: 150ms (micro-interactions like icon fills), --duration-normal: 200ms (standard hover/focus), --duration-slow: 300ms (layout transitions, modals), --duration-slower: 500ms (page-level transitions). Pair with easing tokens: --ease-out: cubic-bezier(0, 0, 0.58, 1), --ease-in-out: cubic-bezier(0.42, 0, 0.58, 1), --ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1). This transforms motion from "whatever the developer felt like" to a coherent kinetic language.
Surface/container tokens. The site has --bg, --bg-elevated, --bg-sunken — a good start. Missing: --surface-card (the default card background, currently inherited), --surface-overlay (modal backdrop, currently hardcoded rgba), --surface-interactive (hover state backgrounds, currently var(--bg-sunken) used inconsistently). Apple's system distinguishes between six surface levels; this product has three.
Border tokens beyond color. --border, --border-subtle, --border-divider define colors, but there's no --border-width-thin: 1px, --border-width-medium: 2px, or --border-radius-card: 16px, --border-radius-hero: 24px, --border-radius-pill: 999px, --border-radius-button: 8px. The border-radius values (6px, 8px, 16px, 24px, 50%, 999px) are used across the codebase without tokens. When the design evolves — say, flattening 24px hero corners to 20px — it requires hunting through every component.
Breakpoint tokens. The media queries use raw values: 390px, 480px, 720px, 767px, 769px, 1024px, 1440px. These should be defined as SCSS/CSS variables or (if using a preprocessor) as named breakpoints: $bp-phone-sm: 390px, $bp-phone: 767px, $bp-tablet: 769px, $bp-desktop: 1024px, $bp-wide: 1440px. The 769px/767px pair (one max-width, one min-width) should be a single token to prevent off-by-one errors.
Interactive state tokens. The hover system is ad-hoc: some elements use translateY(-2px), some use translateY(-1px), some use background tinting, some use border-color changes. Define a vocabulary: --hover-lift-sm: translateY(-1px) (for small UI elements like chips), --hover-lift-md: translateY(-2px) (for cards), --hover-lift-lg: translateY(-4px) (for hero/feature cards), --hover-tint: rgba(0, 0, 0, 0.04) (for icon buttons), --active-press: translateY(0px) scale(0.98) (for click feedback). Currently no :active press states exist anywhere — everything hovers but nothing responds to the click moment itself.
Content width tokens. The reader uses max-width: 604px for headlines and max-width: 670px for body text. The about page uses approximately 604px. These are hardcoded. Define: --content-width-narrow: 560px (tight reading column), --content-width-normal: 640px (standard reading), --content-width-wide: 780px (feature/hero content), --content-width-max: 1400px (app shell ceiling — the missing max-width constraint). This gives every layout decision a named reference point instead of a magic number.
Dark mode token architecture. The current approach uses 41 individual @media (prefers-color-scheme: dark) blocks with hardcoded color overrides per component. A mature system would define all colors as CSS custom properties on :root and swap the entire palette in a single dark mode block. Currently, adding a new component requires remembering to add a dark mode override — this doesn't scale. The fix: define every color used in the app as a --color-* token on :root, then override all tokens in one @media (prefers-color-scheme: dark) { :root { ... } } block. This reduces 41 dark mode rules to 1.

REVOLUTIONARY RECOMMENDATION
Replace the static 6-card reading path with an interactive argument map.
Here's the concept: instead of presenting the six angles as a linear scroll (card 1 → card 2 → card 3 → card 4 → card 5 → card 6), render them as a spatial canvas where the relationships between angles are visually explicit. "What they said" sits at the center. "What we found" branches off with a directional arrow labeled "verified by." "The reframe" branches with "which means." The "Broader view" connects to all three prior cards with "in context of." The result is a visual argument structure — a thinking tool, not just a reading experience.
Why this is transformative: every news app presents information as a feed. Linear. Passive. Scroll down, absorb, move on. The Fourth Angle already has the intellectual infrastructure for something fundamentally different — it decomposes issues into six distinct analytical lenses. But the current vertical scroll flattens that structure into the same linear feed pattern as every other app. The argument map would make the analytical structure visible, navigable, and — critically — shareable. A screenshot of an argument map is worth more than a screenshot of a scrolling article. It's the kind of artifact a policy analyst pins to their wall, a teacher projects in a classroom, a journalist references in a briefing.
The implementation path: on mobile, the map could be a horizontally scrollable node graph (think a simplified Miro or mind-map). On tablet landscape and desktop, it renders as a spatial canvas alongside the text reader — tap a node to read the full angle in a side panel. The fact-graph.json already contains structured relationship data. The reading path progress bar already visualizes the 6-step structure linearly — this just makes the geometry explicit.
This is the feature that would make people say: "I've never seen a news app show me how the argument works, not just what the argument says." It's the visual equivalent of showing your work in mathematics. In a Malaysian media landscape drowning in opinion masquerading as fact, making analytical structure visible is not a feature — it's a mission statement rendered in pixels.
