# Phase 9 — CI: visual, a11y, browser, viewport, perf matrix

**Branch:** `parity/phase-9-ci`

## What ships in this PR

### 1. Playwright e2e matrix
- `playwright.config.ts` with **7 project × viewport combinations**:
  - `chromium-mobile` (Pixel 7), `chromium-tablet` (iPad Pro 11), `chromium-desktop` (1440×900)
  - `firefox-mobile` (Pixel 7 UA + firefox engine), `firefox-desktop` (1440×900)
  - `webkit-mobile` (iPhone 14), `webkit-desktop` (1440×900)
- WebServer: `npx http-server dist -p 4321` against the built `dist/` output (not the dev server — CI asserts production HTML).
- CI: `retries: 2`, `workers: 1` (for stability). Local: defaults.

### 2. Smoke tests (`tests/e2e/`)

**`sidebar-invariant.spec.ts`** — Phase 1 invariant guard. Home page always renders all four canonical `data-section` values (`continue`, `new`, `explore`, `completed`) with numeric `data-section-count` badges. Targets repo-owned `data-*` attributes (safe from stealth rename).

**`ssg-article.spec.ts`** — Phase 8a acceptance. Raw HTML fetch of `/issue/0146` must contain:
- `<article class="ssg-article">`
- `<h1 id="ssg-headline">`
- `og/issue-0146-*w.{avif,webp,jpg}` hero src
- `data-score="NN"` Opinion Shift
- 7 × `data-card-index="N"` sections
- `<time datetime="YYYY-MM-DD">`

**`user-preferences.spec.ts`** — Phases 3, 5, 8.5 gates:
- `prefers-reduced-motion: reduce` → transitions drop to ≈0 ms.
- `forced-colors: active` → Opinion Shift severity label renders as text.
- `localStorage.setItem` throwing → storage-blocked banner appears; dismissable.

**`a11y.spec.ts`** — `@axe-core/playwright` on `/` and `/issue/0146`. Fails on **serious + critical** violations (all rulesets `wcag2a/aa`, `wcag21a/aa`).

### 3. Bundle-size gate
`scripts/check-bundle-budget.mjs` — sums brotli size of every `.js` / `.css` in `dist/_a/` and compares to `scripts/bundle-baseline.json` (established on first run). Fails if delta > +10 KB absolute **and** > +5 % relative — either trim the bundle or update the baseline in the same PR that justifies the growth.

Wired as `npm run check-bundle`. Runs in CI after `npm run build`.

### 4. Font-size gate (shipped in Phase 4, re-used here)
`scripts/check-font-budget.mjs` already in the build pipeline from Phase 4. Fails if total font bytes > 60 KB brotli (ADR-0001).

### 5. GitHub Actions wiring
`.github/workflows/deploy.yml` updated. After build: bundle budget → Playwright browsers install → `npm run e2e` across all 7 projects → upload Playwright HTML report as artifact.

## Lighthouse CI — deferred follow-up

I intentionally did **not** wire Lighthouse CI in this PR. Reasons:

1. **Chromium binary in Codespaces.** Local Lighthouse runs need Chrome; this dev container does not ship one. The `npx playwright install chromium` pull brings Chrome Headless Shell, but Lighthouse CI expects a conventional Chrome profile — they're not quite interchangeable.
2. **CI Chromium is fine** — GitHub Actions' `ubuntu-latest` has Chrome. Lighthouse CI (`@lhci/cli`) can run there, upload reports, enforce thresholds.
3. **Baseline generation is CI-first** per the brief. The first successful run of `lhci autorun` writes report artefacts that PR #15 can check into `docs/cross-browser-parity/baselines/`.

**Recommendation:** ship this PR without Lighthouse CI so the Playwright matrix can land and stabilise. Add `lhci` as a follow-up PR (`parity/phase-9-lighthouse`) that adds an `lighthouserc.cjs`, thresholds per brief (Mobile ≥ 90 Perf / ≥ 95 A11y / ≥ 95 Best / ≥ 95 SEO, Desktop ≥ 95 everything), and the GH Actions step. This keeps the Playwright CI smoke stable while Lighthouse's first-run thresholds get tuned against real CI data.

**Decision needed:** approve ship-as-is, or block this PR until Lighthouse CI also lands?

## Visual snapshots — deferred

Also deferred: `toHaveScreenshot` visual baselines for home / issue / sidebar / forced-colors / reduced-motion / dark mode. Reasons:

1. **Baseline generation environment-dependent** — font hinting and subpixel differ between Playwright's Linux-in-Docker and a Codespace's Linux, so any baseline I generate locally would fail in CI.
2. **Two-run ceremony** — correct practice is: first CI run generates baselines, commit, second CI run compares. Easier as a standalone PR.

**Decision needed:** approve ship-as-is for Phase 9 core; open `parity/phase-9-visual-snapshots` as a follow-up once the matrix is green in CI.

## Running locally

```bash
npm run build              # stealth + sign + generate OG variants
npm run e2e:chromium       # fast feedback loop — just the Chromium projects
npm run e2e                # full matrix (chromium + firefox + webkit)
```

Playwright browsers installed on demand. First run pulls Chromium (~110 MB), Firefox (~85 MB), Webkit (~75 MB).

## Bundle baseline on first run

First-run behaviour of `scripts/check-bundle-budget.mjs`:
- No `scripts/bundle-baseline.json` yet → writes it with the current `total`, exits 0.
- Subsequent runs → compares.

This PR ships the baseline file from its own build output; later PRs that fail the gate must update it with a PR-description justification.
